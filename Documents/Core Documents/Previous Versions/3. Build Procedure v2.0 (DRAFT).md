## 1. Scope & Purpose

Build PLC as a clean, modular, **build-system-free** web project (plain HTML/CSS/JS —
Canvas, SVG, Web Audio; no frameworks). Separate: the **Arcade / PET Terminal hub**; the
**shared engine + dataset logic**; individual **Cartridges** that plug into the engine; and
the neutral **Empty Cartridge** used as the base for new Cartridges. This document is
implementation detail and always defers to the hard rules in Laws v2.0.

## 2. Project Structure — Reality vs. Target

**⚠ Current on-disk reality (2026-07):**
```
Game/
  index.html                     # hub / PET Terminal main menu
  core/                          # hub only: main-menu.js, submenu.js
  assets/  styles/  datasets/    # hub-level shared art / css / csv
  cartridges/
    Asteroid Command/files/      # index.html, script.js, style.css, core/{config,data,audio}.js
    The Aquanaut/files/          # same shape — OWN git repo (see §9)
    Pitstop/files/               # same shape — shell / stubs only
    blank/                       # Empty Cartridge template
```
Two things diverge from the v1.6 blueprint and from the Topic-5 goal:
1. **There is no shared `/core` engine.** Each Cartridge carries its **own** copy of
   `files/core/{config,data,audio}.js`. Core logic is duplicated per Cartridge, not shared.
2. The v1.6 doc described `/core/{engine.js, hud.js, ui.js}` — **these were never built as
   shared modules.** The game loop / HUD / UI live inside each Cartridge's `script.js`
   (Aquanaut's is ~12k lines).

**🎯 Target architecture (Topic 5 — "build once in `/core`"):** extract the shared engine
capabilities below into a real `Game/core/` that Cartridges consume, so new games inherit
them instead of re-copying. This is a **migration**, not a greenfield build — the Aquanaut
is the most complete reference implementation to extract *from*. Do not refactor blindly;
each extraction is its own authorized handoff.

## 3. Shared Engine Capabilities (`/core` target)

These are the Topic-5 capability gaps — the work that belongs in `/core`, built once.

### 3.A Impact Zone system (from §0.9)
Engine primitive:
```js
ImpactZone = {
  id,                                  // 'hose.gas', 'diver', 'tire.fl', 'fall', ...
  approach: 'fall' | 'side' | 'rise',  // direction a Target reaches it (direction-agnostic)
  consequenceType: 'Cumulative' | 'Critical',
  saveWindow: null | <seconds>,        // Critical grace period; drives the countdown UI
  resource: { pool: <poolId>, cost: <n> },  // Cumulative: what reaching it draws down
  terminal: true | false               // does exhausting THIS end the game? (§0.19)
}
Resource = { id, max, onZero: 'gameOver' | { stateChange: <name> } }
```
- **Cumulative pools unify hoses and tires** — they are the *same* mechanism, not two
  (correcting the audit's "two pool semantics"). Both are *N independent sub-targets, each
  with per-target HP, terminal when all are destroyed*: hoses = 3×1 HP (random target
  selection); tires = 4×3 HP (rotating selection). Model them as one grouped-Zone shape with
  `{ count, hpEach, selection: 'random'|'rotating', terminalWhenAllDestroyed: true }`.
- **Not every zero is terminal.** The evaluator must honour each Zone's `terminal` flag. The
  Aquanaut hull is a Cumulative pool with `onZero: { stateChange: 'breach' }` (→ ROV assist),
  **not** `gameOver`. A naive "any pool at 0 → game over" evaluator would wrongly kill the
  player on a routine hull breach.
- **Misses can also draw resources.** Resource depletion is not only from Impact-Zone
  contact — a Miss (mistype) may cost a pool where the Cartridge declares it (Asteroid/
  Aquanaut misfire = 1 hull HP).
- **TFS evaluator:** "check all Zones/resources — has any `terminal` Cumulative pool hit
  zero, or any `terminal` Critical Zone fired?" — generic, replaces bespoke game-over logic.

### 3.B Attack / consequence state machine + kill-screen layer (from Aquanaut)
- One shared state machine: `idle → threat-telegraphed → resolved` (save succeeded /
  consequence applied). **Already built in the Aquanaut** — extract, don't reinvent.
- Kill-screen presentation is **decoupled from the gameplay canvas** (DOM/CSS overlay,
  "dolly-mask crossfade"). Reusable component. In the Aquanaut it is a constructed helmet-POV
  plate with per-creature violent behaviours; terminal words are config-driven and already
  conformed to the ratified set **CRUNCH / SNAP / STUNG / FUGU / CRUSHED**.
  *(Note: the Topics-5 audit text lists the pufferfish word as "POISONED" — that is the
  vision-attack **category**; the terminal **word** is **FUGU**. Fix in the audit.)*
- Random-target selection (which hose breaks) is a generic utility.

### 3.C Partially-ordered command chain + concurrency validator (from NEMS 500)
- **Greenfield — nothing to extend.** Pitstop is a shell (`tireDamageThreshold: 0`,
  pit/tire/fuel "BLOCKED until Overview append", placeholder renderers). No existing
  command-validation does partial ordering.
- Build a **partial-order validator**, not a strict sequence: e.g. Shift Change
  `RCAV → OD → SS → BSEH (must precede CAV SS) → CAV SS`, with `NOTE` insertable any time
  after `SS`. Some steps are order-locked; others free-floating.
- Build it **concurrency-ready from the start** (multiple units in one Shift Change box),
  even before the "batchability" question resolves, to avoid a later rewrite.
- Make **critical-vs-fixable error** a first-class distinction: some wrong-order entries are
  silently-broken-but-recoverable (CAV-before-BSEH looks fixed but isn't), others are hard
  failures — presentation-relevant.

### 3.D Data-driven config (not hardcoded)
- Config/tuning values live in **CSV data files**, so difficulty/tuning changes don't need a
  code round-trip. The Aquanaut proves the pattern — **16 CSVs** (13 in
  `files/datasets/Gameplay/` + 3 in `Game_mechanics/`). *(The audit's "ten CSVs in
  `dataset/Gameplay/`" is stale — the count has grown and the real path is `datasets/`,
  plural.)*
- NEMS 500's tire config (3 hits, tunable), Shift Change frequency ratio, and chain
  definitions should follow the same pattern — currently they are hardcoded placeholders.

### 3.E Asset pipeline (mechanics)
- **Photoreal raster PNG (transparent RGBA)** is the default for characters/creatures/units;
  **SVG for HUD/UI/chrome/geometry**. Animated subjects use discrete state renders with the
  engine cross-fading between them. Gemini prompts must avoid SVG vocabulary.
- File-naming, resolution, folder layout, animation-state and audio-file conventions are
  **game-level**, defined in each Cartridge's Mini-Game Context Packet — not imposed across
  Cartridges (Core Law 9). See the Aquanaut's `Aquanaut_Creature_Asset_Contract.md` as a
  per-Cartridge example, not a system rule.

## 4. Empty Cartridge ("Blank")

Neutral, unthemed base: all core gameplay logic, placeholder visuals (green Targets, yellow
Challenges, blue Command input), no dataset, no theme. Never edited directly. To make a new
Cartridge: duplicate `cartridges/blank/` → apply theme only through Theme Injection Points
(§0.18) → keep core logic intact unless an explicit override is given.

---

## 5. Cartridges

Each Cartridge includes `index.html` (container wired to the engine), `style.css` (theme
visuals), `script.js` (integration + theme logic), an optional `cartridge.json` (id, name,
themeColors, configOverrides), and a **Mini-Game Context Packet** (§0.21) declaring its
Impact Zones, TFS, and conventions. Themes may change visuals/audio/fonts/glow/animation;
they must not change scoring, dataset logic, challenge generation, or engine code.

## 6. Modes, Developer Mode & Access

- **Demo Mode** (default in development): Holodeck disabled; Cartridges are demo cartridges.
- **Publish Mode** (only when instructed): Holodeck enabled, with a "Return to Program"
  button.
- **Developer Mode** (§0.15): detect **Ctrl + Shift + B** → timed password prompt → toggles
  the Blank module / developer testing. Must stay hidden from regular users in non-demo
  builds. Cartridge shells must not block Ctrl + Shift + B at the shell level.
- **Placeholder Access Gate (INTERIM — not canon):** a single client-side "Nerva Beacon —
  Under Construction" login in front of `arcade.html` only. Soft deterrent, **not** a
  privacy control; separate from Developer Mode. Explicitly removable once Nerva Beacon's
  real shell exists.

## 7. Dataset Handling (Excel → CSV → Game)

Maintain a human-friendly Excel master workbook (sheets = Modules / Challenge Sets). Export
relevant sheets as CSVs into the Cartridge's `datasets/` tree; `data.js` loads them at
runtime. CSVs are the **source of truth** (Laws §0.10): mirror exactly, drop what's removed,
verify saves. CSV loading requires an `http://` server (not `file://`) — use the provided
dev-server launcher.

---

## 8. Target / Challenge / Command Behaviour

- **Target** — visual object; always carries an attached Challenge; they move as one unit.
- **Challenge** — plain-language request, rendered relative to the Target (theme-dependent).
- **Command** — correct PowerLine syntax; uses Command Template + Data Pools.
- **Hit / Miss / Impact Zone** — Hit clears the Target; Miss leaves it active (may cost a
  resource); reaching an **Impact Zone** applies that Zone's consequence (§3.A / Overview
  Impact Logic).
- **Glow (themed):** silhouette-following, rotates with the object; lives in the Cartridge's
  theme logic, not the engine.

## 9. Hosting & Deployment *(Topic 7 — with current-state findings)*

**Intended posture (Topic 7):** GitHub Pages, vanilla stack, no build system; **no approval
gate** between a push and Pages going live (Code pushes, Andrew checks after). GitHub's role
is version-history / rollback, **not** a privacy boundary or review gate. A **public repo is
acceptable**, including the Data Sheet CSVs, which Andrew has assessed as not sensitive —
*distinct* from Nerva Beacon's operational content, which has its own governance.

**⚠ Current on-disk reality (verified 2026-07) — the intended posture does not yet match
what exists:**
1. **There is no unified "PLC repo."** The **only** git repo is *inside* one Cartridge —
   `The Aquanaut` (`github.com/darqueshadow/The-Aquanaut---Below-The-Black`). The Arcade
   hub, Asteroid Command, and Pitstop are **not under version control at all**. So "the PLC
   repo auto-publishing the arcade on push" describes something not yet set up.
2. **That repo is private right now.** It returns 404 to anonymous access, and its Pages URL
   404s — i.e. it is not publicly visible and Pages is likely not serving. This contradicts
   the "public is the posture" framing; the current *state* is private. Andrew should
   confirm/flip visibility in GitHub Settings if public is intended.
3. **`main` vs `master` have diverged** — `master` is active (origin/HEAD → master) and
   ahead; `main` is stale (`4109bce`). A "deploy from branch" Pages config must target the
   right branch. **One local commit is unpushed** on `master` (the §0.9 kill-screen work).
4. **Before flipping public,** eyeball the specific CSVs — the data is broader than
   unit/base/command (there are `hospitals.csv`, `inc_numbers.csv`, `ctas.csv`). Nothing
   looks sensitive, but re-confirm the "not sensitive" call covers those exact files.
   **Revisit trigger:** if any Cartridge ever adds real names / real incident data, this
   Law re-opens.

**Recommended near-term actions (need Andrew's go-ahead — do not change repo settings
unprompted):** decide on one PLC repo vs. per-Cartridge repos; bring the hub + other
Cartridges under version control; reconcile `main`/`master` and pick the Pages branch; push
the outstanding commit; then decide visibility.
