## Section 0 — Definitions (Immutable Glossary)

The following terms establish mandatory, unambiguous definitions governing all PowerLine
game construction. They are binding within the scope of the PowerLine Challenge system.
Claude must not reinterpret, alter, or override any definition in Section 0 unless Andrew
explicitly instructs a temporary override. If any instruction appears to conflict with a
definition here, Claude must **pause and request clarification** before proceeding.

### 0.1 Empty Cartridge
The neutral, unthemed base version of the game containing all core gameplay logic and
placeholder visuals. The Empty Cartridge must never be modified directly. Claude must
always duplicate the Empty Cartridge before creating or modifying any Cartridge.

### 0.2 Cartridge
A themed version of the game created from a duplicated Empty Cartridge. Cartridges may
modify appearance, audio, and theme-specific effects but must not modify or replace core
gameplay logic.

### 0.3 Theme
A cosmetic layer defining the visual and audio style of a Cartridge. Themes must never
alter rules, scoring, dataset logic, challenge generation, difficulty progression, Impact
Zone / consequence logic, or any other core system.

### 0.4 Target
A game object representing an actionable item in the playfield. A Target must always carry
an attached Challenge and follow the core movement and lifecycle rules defined in the engine.

### 0.5 Challenge
A plain-language instruction attached to a Target. Challenge generation must follow the
rules defined in the Data Sheet and Overview unless explicitly overridden by Andrew.

### 0.6 Command
The exact PowerLine syntax required to clear a Target. Commands must comply with CAD rules
and reflect the Command Template associated with the Challenge Template.

### 0.7 Clear (Hit)
The state in which a Target is correctly removed from play due to a valid Command input.

### 0.8 Miss
The state in which a submitted Command is invalid. The Target remains active unless cleared
or it reaches an Impact Zone. A Miss may optionally cost a Cartridge-defined resource (e.g.
an integrity pool) where that Cartridge's Mini-Game Context Packet declares it.

### 0.9 Impact Zone  *(generalized — supersedes v1.6 "Impact")*
A **named location a Target reaches if it is not cleared**, approached from any configured
direction (falling, sideways, rising, from below, etc.). A Cartridge declares one or more
Impact Zones. Each Impact Zone declares:

- **consequenceType** — `Cumulative` (reaching it costs one unit of a resource; play
  continues) or `Critical` (reaching it triggers its consequence in a single event).
- **saveWindow** — optional grace period, in seconds, with a visible countdown before the
  consequence applies. Absent by default.

Impact Zone behaviour is engine-governed and must not be altered by Themes (§0.3). Whether
reaching an Impact Zone's failure condition ends the game is governed by §0.19, **not
assumed** — a resource reaching zero is not automatically terminal.

> **This definition retires two v1.6 constructs:** (a) the fall-to-zone-only definition of
> "Impact" (a Target "reaching the lower boundary"), which excluded every non-falling
> Cartridge; and (b) the **"ZONE IMPACT" vs "IMPACT ZONE" dual naming convention**
> (initiator vs receiver). There is now **one** primitive — the **Impact Zone** — and
> Cartridges distinguish cases by `consequenceType`, not by word order.

### 0.10 Data Sheet
The CSV dataset(s) containing command types, units, bases, templates, and data pools. The
Data Sheet is the **source of truth** for game content: Claude must parse and apply it
exactly, mirror it into the game, drop anything removed from it, and verify a save landed
before trusting it. Changes require Andrew's authorization.

### 0.11 Challenge Set
A defined group of Challenge Types used for structured training. Challenge Sets load
according to user selection and must not alter core logic.

### 0.12 Challenge Template
A plain-language template used to construct a Challenge. Must follow the structure defined
in the Data Sheet.

### 0.13 Command Template
The template governing the formation of a required Command. Must align with the
corresponding Challenge Template.

### 0.14 Data Pool
A set of values (Units, Bases, Hospitals, Call Numbers, etc.) used to populate templates.
Values must be drawn exactly as defined.

### 0.15 Developer Mode
A restricted-access mode enabling internal testing, activated via **Ctrl + Shift + B**. It
must remain hidden from regular users in non-demo builds. (Separate from, and not to be
confused with, the interim Placeholder Access Gate — see Build Procedure.)

### 0.16 Blank Dataset Module
A test-only dataset used solely in Developer Mode. Must remain inaccessible during normal
gameplay.

### 0.17 Glow Effect
A contour-following outline applied to themed Targets. Glow must follow the silhouette of
the PNG, rotate with the object, and must never default to bounding-box behaviour.

### 0.18 Theme Injection Points
The exclusive locations Claude may modify when applying a theme: Target appearance,
Challenge appearance, Command UI styling, backgrounds, and optional cosmetic effects. No
other engine components may be changed unless Andrew explicitly authorizes a temporary
override.

### 0.19 Terminal Failure State (TFS)  *(new)*
The condition under which a Cartridge's game ends in failure. **Every Cartridge must define
exactly one.** Under §0.9, a TFS usually **emerges** from Impact Zone state — a Cumulative
resource reaching zero, or a Critical Zone triggering — rather than needing bespoke
game-over logic.

**A resource reaching zero is not automatically terminal.** A Cartridge may define
non-terminal consequences (a degraded state, an assist event) that do *not* end the game.
Each Impact Zone / resource must therefore declare whether its exhaustion is **terminal**.
*(Established by the review: the Aquanaut's dive-bell hull is a Cumulative pool that
depletes to a non-terminal "breach + ROV assist" state, not game-over — so "any pool at
zero ends the game" is false as a blanket rule.)*

### 0.20 PET Terminal  *(new — Topic 1)*
The Arcade hub's permanent visual and interaction identity: Commodore-PET style,
monochrome, phosphor glow, cassette-load prompts. It belongs to the hub, not to any
Cartridge, and is **exempt from Theme Injection Point restrictions** — no Cartridge theme
may override it (see Core Law 6).

### 0.21 Mini-Game Context Packet  *(new)*
The per-Cartridge design document that declares that Cartridge's specifics — including its
**Impact Zones** (each with `consequenceType`, `saveWindow`, and terminality), its TFS, and
its cartridge-level mechanical conventions. A Cartridge's consequence/TFS logic must not be
implemented until its Impact Zones are documented here and authorized by Andrew (Core Law 5).

## Section 1 — Core Laws

1. **Duplicate before building.** Claude must duplicate the Empty Cartridge before creating
   any new Cartridge, and must never modify the Empty Cartridge's core logic directly.

2. **Core stays core.** Claude must never modify core gameplay logic except as defined in
   Overview v2.0 / Build Procedure v2.0. Themes and Cartridges preserve all core objects
   and mechanics unless a Cartridge explicitly and permissibly overrides them.

3. **Themes touch only Theme Injection Points (§0.18).** Themes must never alter rules,
   scoring, dataset logic, challenge generation, difficulty progression, or Impact Zone /
   consequence logic.

4. **Dataset integrity.** The Data Sheet (§0.10) is the source of truth. Mirror it exactly,
   drop anything removed from it, and verify a save actually landed before trusting it.

5. **Impact Zone / TFS authorization gate.** Claude must not implement any Terminal Failure
   State, combat, or consequence logic for a Cartridge until that Cartridge's Impact Zones —
   each with its `consequenceType`, `saveWindow`, and terminality — are documented in its
   Mini-Game Context Packet (§0.21) and authorized by Andrew.

6. **PET Terminal is protected.** The PET Terminal (§0.20) is the hub's permanent identity
   and is exempt from Theme Injection restrictions. No Cartridge theme may override it.

7. **PLC runs standalone (External Staging Context).** PLC must function fully on its own.
   Laws, Overview, and Build Procedure must not describe or depend on external staging
   content (e.g. a corridor, posters, or walk-up sequence within Nerva Beacon). Anything
   depicting entry *into* the Arcade from outside PLC belongs to Nerva Beacon's governance,
   not PLC's.

8. **All repo-bound work routes through Claude Code.** Claude Code is the single point of
   merge into the repository. Other agents (including Claude Design, on pilot) do not commit
   directly; their output is handed off through Claude Code. (See Overview — Team Roles.)

9. **No cross-Cartridge convention imposition.** Claude must not impose one Cartridge's
   naming, folder, or mechanical convention onto another as if it were system-wide, unless
   Andrew explicitly promotes it to Laws/Overview. Cross-Cartridge consistency is a
   nice-to-have, not a requirement.

10. **Modular development.** Separate files for HTML, CSS, JS, images, audio, and data. Do
    not embed large assets unless explicitly instructed for a standalone build.

11. **Modes.** Keep the Holodeck disabled until Publish Mode is engaged. Treat all
    Cartridges applied before Publish Mode as demo cartridges. Do not engage Publish Mode
    unless directly instructed.

12. **Developer Mode** (§0.15) is activated via Ctrl + Shift + B and must remain hidden from
    regular users in non-demo builds.

13. **No unilateral rewrites of governance.** Claude must not unilaterally remove or rewrite
    sections of Laws or Overview — only append or clarify when instructed. A full version
    rewrite (e.g. v1.6 → v2.0) is a deliberate, Andrew-authorized act, not a unilateral edit.

14. **Load the full governance stack before implementing.** For any implementation work,
    load in order: Laws → Overview → Build Procedure → the relevant source files → the
    handoff spec — before writing code.
