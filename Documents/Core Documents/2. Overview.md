## Purpose

The PowerLine Challenge is an interactive, typing-based training game that improves
dispatchers' proficiency with PowerLine (CAD) commands. By offering repeated practice under
real-time pressure, it replaces passive instruction with a hands-on, gamified environment
where players drill core CAD functions until they are second nature.

PLC is a retro-computer-styled arcade — a **PET Terminal** (§0.20) hub that loads
self-contained game **Cartridges**. It must run fully standalone (Core Law 7); its eventual
home as a section inside Nerva Beacon's web shell is Nerva Beacon's concern, not PLC's.

---

## Core Concepts

**Empty Cartridge** — the master template for new games. Holds the core gameplay logic
(canvas, Challenge box, Command box, target spawning/lifecycle, input handling) with
placeholder visuals and no theme or dataset. Cartridges are duplicated from it and themed
through Theme Injection Points; the Empty Cartridge itself is never edited directly.

**Arcade (Main Hub) — the PET Terminal.** Houses the Main Menu and loads Cartridges. Its
identity is the **PET Terminal** (§0.20): Commodore-PET style, monochrome, phosphor glow,
"LOAD" / "INSERT CARTRIDGE" prompts. This identity belongs to the hub and is exempt from
Cartridge theming (Core Law 6).

**Cartridges** — each is a themed game with its own difficulty/dataset/instruction menus,
title and end screens, and menu + gameplay music. Each Cartridge owns a **Mini-Game Context
Packet** (§0.21) declaring its Impact Zones, Resources, TFS, and cartridge-level conventions.

**Holodeck** — a demo mode with a "Return to Program" exit, enabled only in Publish Mode.

**Instructions** — every Cartridge includes an instruction screen and PowerLine command list.

## Gameplay Flow

- Targets spawn on the playfield, each carrying a Challenge.
- The player reads the Challenge and types the correct Command.
- Correct Command → the Target is **Cleared (Hit)**; some Cartridges also fire a projectile.
- Incorrect Command → **Miss**; the Target remains active (and, where a Cartridge declares
  it, a Miss may cost a **Resource** — e.g. the Aquanaut hull).
- If a Target reaches an **Impact Zone** (§0.9) without being cleared, that Zone's
  consequence applies — Cumulative (costs a Resource) or Critical (single-event) — subject
  to any `saveWindow`.
- The game ends when the Cartridge's **Terminal Failure State** (§0.19) is reached.
- Bonus opportunities and boss-style encounters are optional, per-Cartridge mechanics.

Targets may approach an Impact Zone from **any direction** — falling (Asteroid Command),
from the sides (Aquanaut creatures striking the umbilical), etc. The engine is
direction-agnostic; "falling to the bottom" is one configuration, not the rule.

## Impact Logic *(rewritten around the Impact Zone primitive)*

There is **one** impact primitive: the **Impact Zone** (§0.9) — a location a Target reaches
if not cleared. A separate primitive, the **Resource** (§0.22), is a pool depleted by Misses
or consequences (not by a Target reaching a location). A Cartridge declares whatever Zones
and Resources it needs in its Mini-Game Context Packet. The v1.6 fixed list of Impact Types
(Target/Zone/Player/Bonus/Boss Impact) and the initiator-vs-receiver dual naming convention
are **retired** (Bonus/Boss survive only as object/target types) — cases are distinguished by
`consequenceType`, not by word order.

**Consequence types**
- **Cumulative** — reaching the Zone costs one unit of a Resource; play continues. Multiple
  Cumulative Zones may form a group sharing a Resource with a terminal threshold (e.g. "all
  destroyed").
- **Critical** — reaching the Zone triggers its consequence in a single event, optionally
  after a `saveWindow` countdown the player can beat.

**Terminality (§0.19)** — exhausting a Zone or Resource is **not** automatically game-over.
Each declares whether its exhaustion is terminal. The TFS usually emerges from this
configuration; it is not written bespoke per Cartridge.

**Roster mapping (as designed):**

*Impact Zones:*

| Cartridge | Impact Zone(s) | Type | Save window | Terminal condition |
|---|---|---|---|---|
| Asteroid Command | 1 (falling) | Cumulative (shield pool) | none | terminal — shield pool → 0 |
| The Aquanaut | 3 hoses (grouped) | Cumulative | none | terminal — all 3 severed → CRUSH |
| The Aquanaut | 1 diver | Critical | 3 s (type the Command) | terminal — window expires → creature kill |
| NEMS 500 | 4 tires (grouped, 3 hits each) | Cumulative | none | terminal — all 4 blown → game over |

*Resources (not Zones):*

| Cartridge | Resource | Depleted by | onZero |
|---|---|---|---|
| The Aquanaut | dive-bell hull | Misses / misfires | **non-terminal** → breach + ROV assist |

*(The Aquanaut therefore has **two** Impact Zones — hoses and diver, both terminal — plus one
**non-terminal Resource**, the hull. The hull is the §0.19 non-terminal case; it is a
Resource, not a Zone, because nothing "reaches" it.)*

**Terminal words (ratified set):** CRUNCH (shark), SNAP (eel), STUNG (jellyfish), FUGU
(pufferfish), CRUSHED (3rd-hose crush). Config-driven in the kill-screen layer.

**Projectile logic** (optional, per-Cartridge): a projectile travels from an origin (Player,
Zone, Target, or Boss) to a destination and triggers a consequence on collision. Asteroid
Command fires a laser; the Aquanaut fires a sonar pulse; some Cartridges fire nothing.

## Dynamic Dataset Logic

**Universal Formula** (base-posting):
- Challenge: `[Challenge PL] [Unit] [Verb] [Base Name]`
- Command:  `[Command PL] [Unit] [Verb] [Base Number]`
- Example — Challenge: *Assign Unit 2101 to Thorold* → Command: `AP 2101 to 72103`

**Randomization** — one active PowerLine command, one Unit, one Base; Challenge and Command
must share the same Unit/Base pairing. **Validation** — no duplicate Challenge–Command
pairs; every Base Name ↔ Base Number mapping correct.

## Asset Style (system-level; mechanics in Build Procedure)

**Photoreal raster PNG (transparent RGBA) is the project-wide default** for
characters/creatures/units — the intended house style, not an Aquanaut workaround. **SVG is
reserved for HUD / UI / chrome / geometric elements** project-wide. Art-generation prompts
(Gemini) must avoid SVG vocabulary, which triggers contradictory cartoon-vector output.
File-naming, resolution, folder structure, and animation-state conventions are **game-level**
— defined per Cartridge in its Mini-Game Context Packet, not elevated to Laws/Overview
(Core Law 9).
